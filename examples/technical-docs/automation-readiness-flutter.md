# Готовность приложения к автоматизации (Flutter)

**Аудитория:** команда разработки
**Цель документа:** зафиксировать технический gap, из-за которого стандартная автоматизация на adb/UiAutomator2 не работает на этом приложении, и перечислить, что нужно со стороны разработки для перехода к стабильной автоматизации.

---

## Проблема

Приложение построено на **Flutter**. Flutter рендерит весь UI на одном `Canvas` (`FlutterView`), а не через нативные Android-виджеты. Из-за этого:

- **Нативный ввод не доходит до полей.** `adb shell input text`, `input keyevent` и тапы по экранной клавиатуре **игнорируются** кастомными Flutter `TextField` (подтверждено на BUG-001 — поле суммы).
- **UiAutomator2 не видит дерево элементов.** Для нативного драйвера Flutter-экран — это одна большая `View` без доступной иерархии виджетов, поэтому нет надёжных локаторов (id/text/desc).
- Тесты, опирающиеся на координатные тапы, **хрупкие**: ломаются при смене разрешения, локали, размера шрифта.

## Что это значит для тестирования

| Подход | Применимость на текущей сборке |
|--------|-------------------------------|
| Ручное exploratory + adb (dumpsys, screenrecord, tap/swipe) | ✅ Работает, используется сейчас |
| Координатные тапы для автоматизации | 🟡 Возможно, но хрупко — не рекомендуется как основа |
| UiAutomator2 по id/text | ❌ Нет доступных локаторов |
| Ввод текста через `adb input text` | ❌ Игнорируется Flutter TextField |

## Что нужно от разработки для стабильной автоматизации

1. **Test-flavor сборки с Flutter Driver Extension.**
   Включить `enableFlutterDriverExtension()` в отдельном тестовом флейворе (не в production). Это открывает доступ к дереву виджетов и надёжному вводу.

2. **`testID` / `Semantics` на ключевые элементы.**
   Проставить семантические идентификаторы на критичные поля и кнопки: сумма счёта, поле скидки, кнопки подтверждения/удаления, поля карточки пациента. Это даёт стабильные локаторы вместо координат.

3. **Deeplinks для прыжка в нужное состояние.**
   Возможность открыть конкретный экран/состояние по ссылке (например, «форма счёта пациента X»), чтобы тесты не проходили весь UI-путь каждый раз — быстрее и стабильнее.

4. **Тестовый backend с сидером данных.**
   Изолированное окружение с предсказуемыми данными (фикстуры пациентов/счетов) и сбросом состояния между прогонами, чтобы тесты не зависели от случайных данных в проде.

## Целевой стек автоматизации (после внедрения пунктов выше)

- **Appium 2 + WebdriverIO + TypeScript** — оркестрация и e2e-сценарии.
- **Flutter Driver / appium-flutter-driver** — взаимодействие с Flutter-виджетами по `testID`/`Semantics`.
- Координатные тапы — только как fallback для редких нативных диалогов.

## Приоритет внедрения

| Шаг | Эффект | Усилия | Приоритет |
|-----|--------|--------|-----------|
| testID / Semantics на ключевые поля | Стабильные локаторы | Низкие | **H** |
| Test-flavor + Flutter Driver Extension | Доступ к дереву + ввод | Средние | **H** |
| Тестовый backend с сидером | Предсказуемые данные | Средние | M |
